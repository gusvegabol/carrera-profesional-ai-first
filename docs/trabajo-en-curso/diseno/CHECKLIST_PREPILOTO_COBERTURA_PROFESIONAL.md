# Checklist de preparación del piloto de cobertura profesional

> Lista operativa para preparar una prueba experimental. No autoriza por sí sola la ejecución ni crea registros PCS.

## 1. Antes de crear artefactos reales

### Puede prepararlo el agente

- [x] Revisar el [contrato provisional del piloto](../../superpowers/specs/2026-07-19-preparacion-piloto-cobertura-profesional-design.md).
- [x] Confirmar que el flujo usa `PLAYBOOK_COBERTURA_PROFESIONAL_v1_0_0` como orquestador.
- [x] Confirmar que `PLAYBOOK_ENTREVISTA_PROFESIONAL_v1_3_2_IA` se activa solo para la inmersión profunda.
- [x] Revisar las plantillas de entrevistado, mapa, sesión, checkpoint, inmersión y competencia.
- [x] Preparar una copia de trabajo de las plantillas si la convención de identificadores exige ajustes.
- [x] Preparar la matriz de evaluación del piloto.

### Requiere autorización del responsable

- [ ] Autorizar la prueba como experimento metodológico.
- [ ] Identificar a la persona participante o confirmar que el primer caso será el propio responsable.
- [ ] Definir qué información puede conservarse, qué debe excluirse y durante cuánto tiempo.
- [ ] Confirmar el alcance: trayectoria completa, una etapa concreta o una muestra acotada.
- [ ] Elegir la convención de identificadores para este piloto.
- [ ] Confirmar quién revisará la salida y cómo se recogerá su corrección.

## 2. Registro inicial del caso

- [ ] Crear el registro de entrevistado.
- [ ] Crear un mapa de cobertura vacío.
- [ ] Añadir la advertencia de que el mapa será parcial y experimental.
- [ ] Registrar el alcance aprobado y las exclusiones de datos.
- [ ] Verificar que ningún dato sensible se incorpora por defecto.

## 3. Control de cada sesión

- [ ] Preguntar cuánto tiempo tiene la persona y recordar que puede parar.
- [ ] Revisar el mapa previo, si existe.
- [ ] Declarar qué está explorado, qué es candidato, qué quedó parcial y qué sigue pendiente.
- [ ] Realizar la panorámica sin imponer una cronología.
- [ ] Registrar cada responsabilidad diferenciada que pueda merecer una zona propia.
- [ ] Pedir a la persona que elija la zona que desea mirar con más detalle.
- [ ] Aplicar `PATRON_STAR_SIMPLE` a una o dos candidatas como máximo.
- [ ] Activar como máximo una inmersión completa por sesión.
- [ ] No usar una zona, piso, mapa o checkpoint como término técnico en la conversación salvo que la persona lo conozca y lo acepte.
- [ ] No generalizar evidencia de una responsabilidad a otra.
- [ ] Crear checkpoint si la inmersión queda parcial.

## 4. Cierre de cada sesión

- [ ] Separar hechos, evidencias, inferencias, límites y pendientes.
- [ ] Preguntar qué debe corregirse, excluirse o dejarse pendiente.
- [ ] Actualizar el mapa y su advertencia de límite.
- [ ] Registrar candidatas no exploradas y motivo de aplazamiento.
- [ ] Registrar incidencias del modelo o del flujo.
- [ ] Confirmar el siguiente paso posible sin presentarlo como obligación.

## 5. Cierre del piloto

- [ ] La persona ha revisado la salida final.
- [ ] Se han registrado correcciones, rechazos y omisiones voluntarias.
- [ ] Se ha evaluado si el mapa es útil y honesto sobre sus huecos.
- [ ] Se ha evaluado la carga y el control percibidos.
- [ ] Se ha comprobado si una zona parcial puede reanudarse sin inventar contexto.
- [ ] Se ha completado la matriz de evaluación.
- [ ] Se ha emitido una conclusión: continuar, modificar o descartar la propuesta.
- [ ] La conclusión se ha mantenido separada de una decisión metodológica formal.

## Bloqueo actual

El paquete documental puede prepararse, pero no deben crearse artefactos de una persona real ni abrirse la primera sesión hasta recibir las decisiones humanas de la sección 1.
